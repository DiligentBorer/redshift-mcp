# 安全策略

## 支持的版本

本项目仍处于 `0.x` 阶段，仅对**最新发布的版本**提供安全修复。请在上报前先升级到最新版确认问题仍存在。

| 版本 | 是否支持安全修复 |
| --- | --- |
| 0.3.x（最新）| ✅ |
| < 0.3 | ❌ 请先升级 |

## 上报安全漏洞

**请不要通过公开 issue、PR 或讨论区上报安全漏洞**——这会在修复发布前暴露问题。

请使用 **GitHub 私密漏洞上报（Private Vulnerability Reporting）**：

1. 打开仓库的 **Security** 标签页
2. 点击 **Report a vulnerability**
3. 填写漏洞详情并提交

该渠道只有维护者可见，沟通与协作修复全程私密，确认后通过 GitHub Security Advisory 走协调披露。

> 如果仓库的 “Report a vulnerability” 入口不可见，说明私密上报尚未启用；请在 issue 里**不含任何漏洞细节**地留言提醒维护者开启，再走上述流程。

### 上报时请尽量包含

- 受影响的版本 / commit
- 漏洞类型与影响面（如：鉴权绕过、SQL 注入、凭证泄漏、拒绝服务等）
- 可复现的最小步骤或 PoC
- 你设想的攻击场景与潜在影响
- 如有，建议的修复方向

### 响应预期

- **5 个工作日内**确认收到并初步评估
- 确认有效后，私下沟通修复方案与时间线，发布修复版本后再公开披露
- 欢迎在披露时为报告者署名致谢（如你愿意）

## 本项目的安全关注面

本项目是一个独立部署的 **Redshift MCP server**，部署形态决定了它有几处天然的安全敏感面，上报时可重点关注：

- **数据库凭证**：Redshift 连接信息来自 `config.yaml`，绝不应出现在日志、错误消息或工具返回中。
- **Bearer Token 鉴权**：HTTP 传输层用 Bearer token 鉴权（`hmac.compare_digest` 常量时间比较）。任何绕过 `server.path` 鉴权中间件的路径都属安全问题。
- **SQL 只读闸门**：`run_sql` 与声明式 SQL 工具经 `sqlglot` AST 校验，应只允许单条只读查询；任何能绕过只读校验、白名单、或注入非预期 SQL 的方式（含通过插件）都属安全问题。
- **PII 与日志**：绑定参数（可能含 LLM 传入的 PII）只应进受 `sql_audit_level` 闸住的审计通道，不应泄漏进主运行日志。日志中意外出现 SQL 参数 / 凭证属安全问题。
- **依赖漏洞**：传递依赖的已知漏洞由 Dependabot 自动跟踪（见 `.github/dependabot.yml`），通常无需单独上报；但若某依赖漏洞在本项目中可被实际利用，欢迎一并说明利用路径。

## 适用范围

本策略覆盖核心包 `redshift_mcp` 及仓库内自带插件（`plugins/*`）。第三方插件、部署环境（nginx / TLS / systemd 配置）和上游依赖本身的问题不在直接修复范围内，但若经由本项目放大或暴露，欢迎上报。
